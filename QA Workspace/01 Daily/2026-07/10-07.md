Então deve ser contabilizada uma nova citação para o servidor

### CT-003 Diminuir contador de citação ao visualizar citação recebida em cópia de despacho

Dado que um despacho esteja sendo criado

Quando o servidor for envolvido em cópia do despacho

Então deve ser contabilizada uma nova citação para o servidor

E ao visualizar a citação, o contador deve diminuir

Execução Passou?

- [ ]  Não

Evidências de Testes:

### CT-004 Diminuir contador de citação ao visualizar citação recebida em menção de comentário

Dado que um comentário esteja sendo criado

Quando o servidor for mencionado no comentário

Então deve ser contabilizada uma nova citação para o servidor

E ao visualizar a citação, o contador deve diminuir

Execução Passou?

- [ ]  Sim
- [ ]  Não

Evidências de Testes:

---

### Critérios de Aceite

- O sistema deve contabilizar uma nova citação para o servidor quando este for envolvido como responsável na criação de um documento.
- O sistema deve contabilizar uma nova citação para o servidor quando este for envolvido em um despacho.
- O sistema deve contabilizar uma nova citação para o servidor quando este for envolvido em cópia de despacho.
- O sistema deve contabilizar uma nova citação para o servidor quando este for mencionado em um comentário.
- O contador de citações deve ser diminuído somente após a visualização da citação pelo servidor.
- O comportamento de contagem e diminuição do contador deve ser consistente em todos os pontos de envolvimento do servidor (criação de documento, despacho, cópia de despacho e menção em comentário).

## SGV-9959 - Casos de Teste — Assinatura Sequencial

### CT-005 Cancelar assinaturas pendentes ao recusar assinatura em fluxo sequencial

Dado que exista uma solicitação de assinatura sequencial com múltiplos servidores

Quando um servidor recusar sua assinatura

Então as assinaturas pendentes dos servidores posteriores devem ser canceladas

Execução Passou?

- [ ]  Sim
- [ ]  Não

Evidências de Testes:

### CT-006 Permitir nova solicitação de assinatura para servidor após cancelamento

Dado que a assinatura de um servidor tenha sido cancelada após recusa em um fluxo sequencial

Quando uma nova solicitação de assinatura for criada para esse servidor

Então a nova solicitação deve ser exibida com status "Pendente" no drawer de solicitação de assinaturas

Execução Passou?

- [ ]  Sim
- [ ]  Não

Evidências de Testes:

### CT-007 Manter histórico de recusa e cancelamento no evento da solicitação anterior

Dado que uma assinatura tenha sido recusada e as demais assinaturas da solicitação tenham sido canceladas

Quando eu acessar o evento da assinatura referente a essa solicitação

Então o status deve permanecer exibido como "Recusado"/"Cancelado", preservando o histórico

E esse registro não deve ser alterado por uma nova solicitação criada posteriormente

Execução Passou?

- [ ]  Sim
- [ ]  Não

Evidências de Testes:

### CT-008 Assinar normalmente em nova solicitação após cancelamento anterior

Dado que exista uma nova solicitação de assinatura pendente para um servidor cuja assinatura anterior foi cancelada

Quando o servidor assinar a nova solicitação

Então a assinatura deve ser concluída normalmente, sem erros ou bloqueios

Execução Passou?

- [ ]  Sim
- [ ]  Não

Evidências de Testes:

---

### Critérios de Aceite — Assinatura Sequencial

- Ao recusar uma assinatura em um fluxo sequencial, todas as assinaturas pendentes posteriores devem ser canceladas.
- Após o cancelamento, deve ser possível criar uma nova solicitação de assinatura para os mesmos servidores.
- A nova solicitação deve poder ser assinada normalmente, sem bloqueios decorrentes do cancelamento anterior.
- O evento da solicitação anterior deve preservar o histórico de recusa/cancelamento, independentemente de novas solicitações criadas.
